## Product idea

RailCross is railway-crossing-aware navigation. It aims to make traffic delays at level crossings explainable by predicting whether a gate is `OPEN` or `CLOSED`, estimating the remaining closure duration, and exposing the result in a map/route-comparison UI.

It fuses three evidence types:

1. Traffic-derived signals inspired by traffic-aware routing APIs: static duration, traffic duration, approach speeds, queue estimates, congestion age, and rolling traffic trends.
2. Train-schedule proximity, used as an optional Bayesian prior at inference time.
3. Geofenced crowdsourced gate reports, intended as independent consensus evidence.

The project is currently an ML prototype and decision-support demo. It must not be represented as rail-control software, a verified gate-state service, or a real-world accuracy claim.

## ML data and evaluation scope

Training data is generated by an event-driven synthetic simulator. It produces railway closures and six non-rail hard-negative traffic conditions: road accidents, market days, construction zones, school zones, signal failures, and weather flooding. This is intentional because verified real gate labels are not yet available.

The split is chronological by complete simulated event: 70% train, 15% validation, 15% test. This avoids event-row leakage. Threshold selection occurs on validation data.

Latest reproducible synthetic-test metrics in `artifacts/model_evaluation.json`:

## Review focus areas

1. Is the ML framing technically honest and portfolio-ready for synthetic data?
2. Does the simulator create a meaningful generalisation challenge, or are there label/feature shortcuts?
3. Are the classifier, schedule prior, crowdsourced evidence, and reopening estimator combined coherently?
4. What evaluation additions are needed before a real-world pilot?
5. How should Cloudflare D1 handle high-volume concurrent reports? Consider Durable Objects for per-crossing serialization, Queues for raw-report ingestion, idempotency, rate limits, and aggregate writes.
6. What should the CCTV architecture be? Assess edge sampling, detector/keypoint classifier, temporal tracking, confidence calibration, privacy/redaction, and human-review workflow.
7. How should Google Routes polling be made cost-aware? Assess adaptive polling, cache/freshness windows, overlap deduplication, budget circuit breakers, quotas, and escalation only for likely closures.
8. Find implementation gaps that might prevent this from being a robust full-stack ML project.

## Current known limitations

- No independently labelled real-world gate data yet.
- Crowdsourced-reputation lifecycle and strong anti-abuse controls are incomplete; the existing report route is a prototype.
- D1 is currently used directly for reports; Durable Objects and Queues are recommended before a high-concurrency pilot.
- CCTV code is a stub, not an implemented visual classifier.
- Routes poller estimates some queue features heuristically and has no production scheduler/budget service.
- Demo history/monitoring are clearly marked but still need replacement with operational measurements during a real pilot.
- Push notifications use prototype handling and need a real VAPID/key-management and dispatch service.
